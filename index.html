Introduction

Employee Service Design Kit

Ready-to-use shell templates for Employee Service products. Start prototyping directly in Cursor or your favorite AI-powered editor. Iterate fast, share with your team, and hand off production-ready code to engineers.

Why use Employee Service shells?

Full repositories contain hundreds of thousands of lines of code. This creates context overload for AI tools—they lose focus and produce inconsistent results. You waste time managing context, fixing errors, and burning through tokens inefficiently.

Employee Service shells give you just what you need: a complete Zendesk navigation framework. Focus entirely on prototyping your feature. The shell includes the header with product tray, sidebar navigation, and all underlying structure. Build fast, test immediately, and gather feedback in real time.

What is the shell?

The shell is the complete Zendesk navigation framework—everything except your feature. It includes the header, sidebar, and structure built with Flora or Garden, depending on the product area you're working in.

Your content goes in the <Main> area. The header, sidebar, and navigation stay fixed. Build your tables, forms, and workflows inside Main. The shell handles everything else.

Benefits

Employee Service shells provide:

  • Speed: Start prototyping in minutes.
  • Consistency: Every shell matches Zendesk's production interface.
  • AI-friendly: Optimized codebases save tokens and time.
  • Production-ready: Clean, accessible code ready to hand off.
  • Shareable: Host prototypes on GitHub Pages.

Who can use this

Employee Service shells are for anyone who needs to prototype or communicate product ideas:

  • Designers: Prototype features, test flows, and validate designs.
  • Product managers: Build interactive demos to align stakeholders.
  • Anyone with an idea: Bring Employee Service features to life.
Setup

Getting started

Set up your shell template and start prototyping in 5 steps. No coding experience required.

What you'll need

Before you start, make sure you have:

New to vibe coding?
Check out two great resources: Cursor Enablement Program by Andrii Lytvynko and Vibes to Production by Abi Iyer.
1

Clone the design kit

ONE TIME ONLY
Terminal
$mkdir -p ~/zendesk/design-kits ~/zendesk/prototypes
$cd ~/zendesk/design-kits
$git clone git@github.com:zendesk/employee-service-design-kit.git
2

Create a project folder

Terminal
$mkdir ~/zendesk/prototypes/my-new-project

Each prototype lives in its own folder.

3

Open in Cursor

  1. Click File → Open Folder
  2. Navigate to ~/zendesk/prototypes/my-new-project
  3. Click Open
4

Run the setup script

Open your terminal in Cursor or any tool you're using, and run this command. It copies the template, installs libraries, sets up git, and configures everything automatically:

Terminal
$~/zendesk/design-kits/employee-service-design-kit/scripts/setup-here.sh itam-shell

Takes about 30 seconds. Progress messages appear as files copy and dependencies install.

5

Start the dev server

Launch the local web server. It watches for changes and auto-refreshes:

Terminal
$npm run dev

Building with AI

The setup runs entirely in Cursor — no context switching between tools.

Two AI tools, one editor:

  • Cursor Agents: Handles all UI work. Reads Figma components via MCP, understands Flora patterns, and builds what you describe. Perfect for styling, layouts, and visual tweaks with instant side-by-side diffs.
  • Claude Code (in Cursor terminal): Takes complex tasks off your hands. Generates mock data, refactors state management, searches repos, and connects to external tools via MCP — all autonomously.

Model Context Protocol powers both:

  • Figma MCP: Both tools pull design tokens, variables, and auto-layout directly from Figma. No pixel guessing. Claude Code can even push rendered UI back as editable Figma layers to keep design files synced.
  • GitHub MCP: Branch creation and PR staging happen natively without switching apps.

Result: Cursor Agents perfects the look, Claude Code handles the logic, and MCP keeps code and design in sync. Prototyping that used to take hours now takes minutes.

Process

Workflow

Build prototypes with natural language — no coding required.

1. Build with AI

Describe what you want in plain English. Cursor Agents handle the code.

Two approaches:

  • Figma MCP: Cursor reads components directly from your Figma files
  • Copy-paste: Copy React code from Figma Dev Mode and paste into prompts

2. Iterate

Changes appear instantly. Refine until it feels right.

  • Ask Cursor to adjust layouts, spacing, or component choices
  • Browser auto-refreshes on every save
  • Test flows and edge cases as you build

3. Share

Show your work in meetings or send a live link.

  • Screen share: Present localhost in meetings
  • Publish: Create a shareable link (see next step)

4. Publish

Deploy to GitHub Pages for stakeholder review.

Terminal
$ ./publish.sh

Creates a private repo and live site. Configures npm authentication automatically.

5. Update

Push changes to your live site.

Terminal
$ ./push-update.sh

Bumps version, updates CHANGELOG, rebuilds site.

6. Quality checks

Run automated checks before developer handoff.

IN PROGRESS

These scripts are being refined and will convert into Claude Code skills.

Compliance check

Code
$ ./check-compliance.sh

Translation check

i18n
$ ./check-translations.sh

Accessibility audit

a11y
$ ./audit-accessibility.sh

7. Developer handoff

Create the final package for engineering.

Terminal
$ ./finalize-handoff.sh

Runs all checks, generates docs, bumps to v1.0.0.
