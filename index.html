INTRODUCTION

Employee Service Design Kit

Ready-to-use shell templates for Employee Service products. Start prototyping in Cursor or your favorite tool. Iterate fast, share with your team, and hand off clean code to engineers.

Why use Employee Service shells?

Full repositories contain hundreds of thousands of lines of code. This creates context overload for AI tools. They lose focus and produce inconsistent results. You waste time managing context, fixing errors, burning through tokens inefficiently, and waiting on slow AI response times.

Employee Service shells give you exactly what you need: a complete Zendesk navigation framework. Focus on prototyping your feature. The shell includes the header with product tray, sidebar navigation, and all underlying structure. Build fast, test immediately, and share for feedback.

What is the shell?

The shell is the complete Zendesk navigation framework, everything except your feature. It includes the header, sidebar, and structure built with Flora or Garden, depending on the product area.

Your content goes in the <Main> area. The header, sidebar, and navigation stay fixed. Build your tables, forms, and workflows inside Main. The shell handles the rest.

Benefits

Employee Service shells give you:

  • Speed: Start prototyping in minutes.
  • Consistency: Every shell matches Zendesk's production UI.
  • AI-friendly: Optimized codebases maximize prompt efficiency, saving tokens, cost, and time.
  • Production-ready: Clean, accessible code that's ready to hand off.
  • Shareable: Host prototypes on GitHub Pages.

Who can use this

Use Employee Service shells to prototype or communicate product ideas:

  • Designers: Prototype features, test flows, and validate designs.
  • Product managers: Build interactive demos to align stakeholders.
  • Anyone with an idea: Bring Employee Service features to life.
SETUP

Getting started

Set up your shell template and start prototyping in five steps. No coding experience required.

What you'll need

Before you start, make sure you have:

New to vibe coding?
Check out two great resources: Cursor Enablement Program by Andrii Lytvynko and Vibes to Production by Abi Iyer.
1

Clone the design kit

ONE TIME ONLY
Terminal
$mkdir -p ~/zendesk/design-kits ~/zendesk/prototypes
$cd ~/zendesk/design-kits
$git clone git@github.com:zendesk/employee-service-design-kit.git

Open Terminal (press Cmd + Space, type "Terminal", hit Enter), copy the script above, paste it in, and press Enter.

This creates your workspace folders and downloads the design kit. You'll use this folder structure for every prototype.

Already cloned the design kit?

Terminal
$cd ~/zendesk/design-kits/employee-service-design-kit
$git pull

Run these commands to get the latest templates and scripts.

2

Create a project folder

Terminal
$mkdir ~/zendesk/prototypes/my-new-project

Skip Terminal if you want. Just create the folder in Finder. Go to zendesk/prototypes and add your project folder.

Each prototype gets its own folder.

3

Open in Cursor

  1. Select File → Open Folder or press Cmd + O
  2. Navigate to ~/zendesk/prototypes/my-new-project
  3. Select Open
4

Run the setup script

Terminal
$~/zendesk/design-kits/employee-service-design-kit/scripts/setup-here.sh

Switch to Cursor's integrated terminal now. Press Ctrl + ` (backtick) or select Terminal → New Terminal.

Copy the script, paste it in Terminal, and hit Enter. Pick your template from the wizard. This copies template files, installs dependencies, and sets up git. Takes about 30 seconds.

If you run into issues with this step, visit Troubleshooting.

5

Start the dev server

Terminal
$npm run dev
VITE v8.1.0 ready in 245 ms
➜ Local: http://localhost:5173/
➜ Network: use --host to expose

The dev server starts and shows a local URL (something like http://localhost:5173/). Open it in Cursor's browser.

Changes show up automatically when Cursor Agents finish. No refresh needed.

You're done!

Your prototype is now running. Everything you see is:

  • Using Flora and Garden libraries
  • Built to match Zendesk's production UI
  • Pre-filled with content in the <Main> area

Working with Cursor Agents and Claude Code

Use Cursor Agents and Claude Code together in one window. Cursor Agents build the UI and interactions. Claude Code handles complex workflows, generates mock data, refactors code, and connects to external tools. Cursor Agents build what you see, and Claude Code plans how it works.

SKILLS

Skills

Skills are shortcuts you run right in your editor. Type /name in Claude Code or @name in Cursor, and the skill handles a task for you. Every prototype comes with them built in, so there's nothing to install.

Prototyping
match-repo

What it does

This skill grounds your prototype in the real production code. It asks which product area and which page you're prototyping, then reads the target repo straight from GitHub, including its dependency versions and data shapes. It saves what it learns to a file the AI follows as it builds. Your prototype uses the right versions, the right components, and the right data, so engineers can reuse the code instead of rebuilding it.

When to use

Run this first, right after you bring a design over from Figma. It sets up the context that later steps rely on.

Invoke it

Claude Code
>/match-repo
Cursor
>@match-repo

What you get

One file, .claude/target-context.md. Later steps read it to keep your prototype in line with the real repo. It also fills in the README's version line and any ownership details it can pull from the target repo.

publish

What it does

This skill publishes your prototype for the first time. It creates a private GitHub repo that only Zendesk employees can see, pushes your code, and sets up a live GitHub Pages link you can share.

When to use

Run this when you're ready to share your prototype with the team. Use it the first time only. For later changes, use /publish-update.

Invoke it

Claude Code
>/publish
Cursor
>@publish

What you get

A private repo and a live link you can share. It also wires up the deploy workflow and npm authentication for you.

publish-update

What it does

This skill ships a new version of an already-published prototype to its live site. It reads what changed, suggests a version bump, and refreshes the README's version line — the date and the production repo it's based on — from your changes. There's no changelog to write by hand.

When to use

Run this any time you've made changes worth sharing, after you've published once with /publish.

Invoke it

Claude Code
>/publish-update
Cursor
>@publish-update

What you get

A bumped version, a refreshed README version line, and an updated live site.

repo-check

What it does

This skill checks that your prototype still matches the real target repo. It compares your work against the context /match-repo created, looking at versions, dependencies, components, and data shapes. It reports only where the two differ. It doesn't review your code quality. That stays with the engineer.

It also checks that the repo follows the kit's standard — the README structure, the two allowed statuses, and the expected files — so nothing surprises the engineer at handoff.

When to use

Run this before you hand off to developers. Run /match-repo first, since this check compares against the context it creates.

Invoke it

Claude Code
>/repo-check
Cursor
>@repo-check

What you get

A list of the gaps between your prototype and the target repo, grouped by versions, dependencies, components, and data shapes, plus any places the repo drifts from the kit standard.

handoff

What it does

This skill packages your finished prototype for engineers. It reads your real code and writes a single HANDOFF.md that spells out what to reuse, what's throwaway scaffolding, and how the feature wires up to production data, so an engineer can pick it up from that one file. It also marks the README Ready for development, tags a v1.0.0 release, and runs a production build.

When to use

Run this when the prototype is done and you're ready for developers to pick it up. Run your quality gates first: /repo-check, /localize, and /a11y-check.

Invoke it

Claude Code
>/handoff
Cursor
>@handoff

What you get

A single, self-contained HANDOFF.md written from your real code, the README marked Ready for development, a v1.0.0 release, and a production build.

Localization
localize

What it does

This skill prepares the final text of your prototype for the Zendesk translation process. You choose exactly which strings need translation. That means only your new text, not the template. It creates a Rosetta-compliant i18n/en-us.yml file that an engineer can drop into their repo.

When to use

Run this once your content is final. That usually means you've explored options, shared with your team, iterated with engineering and product, and reviewed with content design. You can run it any time and update later, so there's no wrong moment.

Invoke it

Claude Code
>/localize
Cursor
>@localize

What you'll do

1

Tell it which component or page has the final text.

2

It shows a numbered list of the strings it found. Technical strings are filtered out for you.

3

You pick which ones need translation (e.g. 1, 2, 4, or all, or none).

4

It generates the YAML file. This includes real values and translator descriptions, plus placeholders for the parts your engineer decides.

5

It validates the file, commits it, and asks before pushing to your repo.

What you get

A i18n/en-us.yml:

i18n/en-us.yml
title: "PLACEHOLDER_FILE_TITLE"
packages:
  - PLACEHOLDER_PACKAGE

parts:
  - translation:
      key: "txt.PLACEHOLDER.0001"
      title: "Drawer title - Noun phrase - Heading of the edit-columns drawer."
      value: "Edit columns"
      screenshot: "PLACEHOLDER_SCREENSHOT"

What the engineer finalizes

The file is a ready-to-use draft, not a broken one. An engineer fills in a few placeholders that vary by team:

Placeholder Who fills it Why
PLACEHOLDER_FILE_TITLE Engineer Document title, per team convention
PLACEHOLDER_PACKAGE Engineer Real Rosetta package name
txt.PLACEHOLDER.NNNN Engineer Key naming varies by team
PLACEHOLDER_SCREENSHOT Engineer Screenshot captured & named after the final key

Everything else is ready to go. The string values, translator descriptions, formatting, pluralization, and interpolation all follow Zendesk standards and are String-Sweep-ready.

localize-update

What it does

When content designers change some wording later, this skill updates only the strings you already prepared. It replaces just the value of existing entries. Your keys, descriptions, and structure stay the same, so you don't have to redo the whole process.

When to use

Run this after a content revision, once you've already run /localize.

Invoke it

Claude Code
>/localize-update
Cursor
>@localize-update

What you'll do

1

Tell it what changed. You can point it to a component to re-scan, or give it the old and new text directly.

2

It shows the matched entry before changing anything.

3

It swaps in the new text, re-validates, commits, and asks before pushing.

Note: your engineer will need to recapture any screenshots for the strings you changed.

Accessibility
a11y-explore

What it does

This skill helps you pick the right component before you build. When you're choosing a component or sketching an interaction, it recommends the correct Garden or Flora component and points out the accessibility implications first. It only gives advice. It never edits your code.

When to use

Run this early, while you're exploring. Use it any time you're deciding how to build something.

Invoke it

Claude Code
>/a11y-explore
Cursor
>@a11y-explore

What you get

Conversational guidance only. No files.

a11y-check

What it does

This skill reviews a component for accessibility before you ship, built on the zendesk-accessibility plugin. You choose which component or interaction to check, not the whole shell. It checks the code against WCAG 2.2 AA and sorts what it finds into Critical, Major, and Minor. It explains each issue in plain language, then fixes it in place with your approval, using the right Garden or Flora components. It confirms each fix with your project's build.

When to use

Run this near the end, once your content and behavior are final.

Invoke it

Claude Code
>/a11y-check
Cursor
>@a11y-check

What you get

A plain-language list of what it found, and the fixes applied to your code once you approve them.

PROCESS

Design to Development

An experimental AI workflow that takes an idea from design to production, grounded in the real codebase from the first prompt.

What this is

This is an experimental workflow that uses AI to take an idea from design all the way to production. It brings together Cursor, Claude Code, and a set of skills that ground your prototype in the real Zendesk codebase.

What this is not

This isn't about contributing to production code. That's a separate topic, covered under Contribute to Code. Here, the goal is a realistic prototype that engineers can pick up with confidence.

First, connect the MCPs

This workflow depends on a set of MCP (Model Context Protocol) servers that connect your AI tools to Zendesk's systems. Connect them to both Claude Code and Cursor before you start. This is the one step you can't skip.

You'll find them in the AI Gateway.

This workflow uses these MCP servers:

Your engineering pair

Cursor gives you more than a plain editor like VS Code. The AI agents are built in, they can see your code and your design context, and you iterate right in place.

A simple way to think about it: Cursor Agents are your front-end engineer, and Claude Code is your back-end engineer. Cursor builds the UI. Claude Code cross-checks everything around your product, including real data, real repos, and design patterns for specific interactions.

1

Prototyping with vibe coding

Before you type your first prompt to Cursor, run /match-repo. This is the most important step on this page. It grounds the prototype in your target repo, so you match real versions and components from the start, and it makes the rest of the workflow far smoother.

Don't tackle everything at once. Rather than handing Cursor a whole-screen Figma frame, work at the component or page-section level. You control accuracy more easily, and you can focus on interaction details and accessibility, not just matching the Figma. Use the Figma URL through the figma MCP server so the prototype stays accurate.

Example prompt
Build the filter bar from this frame: @https://www.figma.com/design/abc123?node-id=45
Use Garden. A primary <Button>, a <Combobox> in its default state, and a small ghost <Button> for reset.

Then iterate. Share your work, gather feedback, and make changes.

2

Publishing and updating

When you're confident in your explorations, or you need to share with a PM or stakeholders, run /publish. It creates the repo and a live website with your prototype on it, and sets up a Jira-first README that opens with the Jira link, a status of Work in progress, and the demo link, with ownership details right there in the file.

After you publish, run /publish-update whenever you make changes. It pushes to the repo and rebuilds the live site.

There's no versioning at this stage, so you can keep making changes and pushing commits without confusing anyone.

3

Quality checks before handoff

Think of this as getting your work dev-ready, the way you'd prepare Figma files for engineering, and then going a step further. Run these quality checks before you hand off. The order below is a good default, but the sequence is your call.

  • /repo-check: Confirms the prototype still matches the real target repo across versions, dependencies, components, and data shapes, and that the repo follows the kit standard.
  • Localization (/localize, /localize-update): Prepares your final text for Zendesk's translation process.
  • Accessibility (/a11y-explore, /a11y-check): Gets interactions and components to WCAG 2.2 AA, exploring choices early and reviewing before handoff.
4

Handoff

When you're confident the work is ready for engineers, run /handoff. It reads your real code and writes a single, self-contained HANDOFF.md — one guide an engineer reads to know what to reuse, what's throwaway scaffolding, and how to wire it to production data. It also flips the README status to Ready for development, marks it as v1.0.0, and builds it. The result is a clean package for the engineering team.

Where the workflow is still evolving

This is an experiment, and some essential pieces are still in discovery:

  • Collaboration: Having more than one person work on the same project.
  • Multiple explorations: Comparing several versions side by side, the way you would in Figma.
  • Content reviews: Making reviews with the content team easier, since today they're still manual.
  • The skills themselves: Testing this first version in real projects to find the weak spots and improve them.
HELP

Troubleshooting

Common issues and solutions for setting up and using Employee Service shells.

npm authentication error

If npm install fails with authentication errors, authenticate with Zendesk's npm registry. You only need to do this once per machine.

1

Add registry to ~/.npmrc

~/.npmrc
@zendesk:registry=https://zdrepo.jfrog.io/artifactory/api/npm/npm/
@zendesk-ui:registry=https://zdrepo.jfrog.io/artifactory/api/npm/npm/

Create this file in your home directory if it doesn't exist.

2

Authenticate with Zendesk LDAP

Terminal
$npm login --registry=https://zdrepo.jfrog.io/artifactory/api/npm/npm/ --scope=@zendesk
$npm login --registry=https://zdrepo.jfrog.io/artifactory/api/npm/npm/ --scope=@zendesk-ui

Once you're authenticated, run the setup script again.
